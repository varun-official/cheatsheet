Core Concepts

Topic

  • Named stream of messages
  • Like a DB table but append-only
  • Split into partitions
  • Multiple consumers can read

Partition

  • Ordered, immutable log
  • One message → one partition
  • Partition count = max parallelism
  • Same key → same partition always

Offset

  • Position number in a partition
  • Starts at 0, increments forever
  • Stored in __consumer_offsets
  • Survives consumer crashes

Broker

  • A Kafka server / machine
  • Holds partition data on disk
  • Cluster = multiple brokers
  • Only need 1 address to connect

Partition + Consumer Group Rules
R1
One partition → one consumer within a group at any time. Never shared.
R2
Instances = Partitions is the sweet spot. Extra instances sit idle. Too few = one instance handles multiple partitions.
R3
Different groups each get a full independent copy of all messages. Same group = work is divided.
R4
Partition count = max parallelism. Adding instances beyond partition count gives zero throughput gain.
R5
Set partitions generously upfront (6, 12, 24). Increasing later requires careful planning.

Producer
from confluent_kafka import Producer
import json

producer = Producer({
    'bootstrap.servers': 'localhost:9092',
    'acks': 'all',            # safest
    'enable.idempotence': True  # no duplicates on retry
})

def on_delivery(err, msg):
    if err: print(f'Failed: {err}')
    else: print(f'offset {msg.offset()}')

# add to buffer (non-blocking)
producer.produce(
    topic='orders',
    key='user-42',          # same key → same partition
    value=json.dumps(order),
    callback=on_delivery
)
producer.poll(0)            # trigger callbacks
producer.flush()            # wait for all — always at end

Consumer
from confluent_kafka import Consumer, KafkaException

consumer = Consumer({
    'bootstrap.servers': 'localhost:9092',
    'group.id': 'order-processor', # team name
    'auto.offset.reset': 'earliest',
    'enable.auto.commit': False,      # manual = safer
    'partition.assignment.strategy': 'cooperative-sticky'
})

consumer.subscribe(['orders'])

try:
    while True:
        msg = consumer.poll(timeout=1.0)
        if msg is None: continue
        if msg.error(): raise KafkaException(msg.error())

        order = json.loads(msg.value())
        process(order)           # your logic
        consumer.commit(msg)    # commit AFTER processing
finally:
    consumer.close()            # always close cleanly

Consumer group partition assignment

ScenarioResult
Instances = PartitionsPerfect — 1 each
Instances < PartitionsSome get 2+ partitions
Instances > PartitionsExtra instances idle
Instance crashesKafka auto-rebalances

Rebalance strategies

StrategyBehaviour
eager (old)ALL consumers stop then reassign — causes downtime
cooperative-stickyOnly moved partitions pause — zero downtime
→ Always use cooperative-sticky in production
Replication + ISR

Leader

  • Main copy of partition
  • All reads + writes go here
  • One per partition at a time
  • Elected from ISR on crash

Follower

  • Silent copy of partition
  • Constantly syncs from leader
  • Does not serve reads/writes
  • Becomes leader if in ISR

ISR

  • In-Sync Replicas list
  • Followers fully caught up
  • Lag behind → removed from ISR
  • Only ISR members can lead
Replication FactorBrokers neededCan survive
11Nothing — data lost on crash
221 broker crash
3 (recommended)32 brokers crashing simultaneously
Delivery Guarantees + Dead Letter Queue

At most once

commit → process

  • Crash after commit = message lost
  • Use for: logs, click tracking

At least once

process → commit

  • Crash before commit = processed twice
  • Use for: most app events

Exactly once

atomic transaction

  • Never lost, never doubled
  • Use for: payments, inventory
# Exactly-once producer config

Message Retention + Log Compaction
Retention — delete messages by time or size. Default: 7 days. Messages stay whether read or not — this enables replay.
# Time-based retention (default 7 days)
config={'retention.ms': '604800000'}

# Size-based
config={'retention.bytes': '1073741824'}  # 1GB

# Keep forever (careful with disk!)
config={'retention.ms': '-1'}

# Common retention by use case
# Logs:     3 days
# Orders:   30 days
# Activity: 7 days
# Audit:    1 year or forever
Log Compaction — keep only latest value per key. Use for user profiles, prices, config — anything where history doesn't matter, only current state does.
# Enable compaction
config={'cleanup.policy': 'compact'}

# Before:  user-1:Mumbai  user-1:Pune  user-1:Bangalore
# After:   user-1:Bangalore  (only latest kept)

# Delete a key — send null (tombstone)
producer.produce(
    topic='user-profiles',
    key='user-3',
    value=None  # ← tombstone = delete on compaction
)
Schema Registry
Schema Registry enforces message format contracts between producers and consumers. Prevents breaking changes. Uses Avro / Protobuf / JSON Schema.
order_schema = """
{
  "type": "record",
  "name": "Order",
  "fields": [
    {"name": "order_id", "type": "int"},
    {"name": "amount",   "type": "float"},
    {"name": "item",     "type": "string"}
  ]
}
"""

registry = SchemaRegistryClient({'url': 'http://localhost:8081'})
serializer = AvroSerializer(registry, order_schema)
producer.produce(topic='orders', value=serializer(order))

Kafka Streams (Faust)

import faust

app = faust.App('processor',
    broker='kafka://localhost:9092')

orders = app.topic('orders')
high_val = app.topic('high-value')

@app.agent(orders)
async def process(stream):
    async for order in stream:
        if order['amount'] > 10000:
            await high_val.send(value=order)

Backpressure + Lag

Consumer slower than producer → lag grows in partition.

  • Add more consumer instances (best)
  • Optimize processing code
  • Increase partition count
  • Monitor lag daily
Sends 1000/s · Handles 200/s
Lag +800/s → add instances
